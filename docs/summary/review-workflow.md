# 审核流程模块

## 一、模块概述

### 1.1 定位和作用

审核流程模块是成本分析管理系统的核心业务流程模块，负责成本分析的审核管理。通过严格的审核机制，确保成本分析的准确性和合规性，审核通过的成本分析可设为标准成本供后续引用。

### 1.2 功能范围

- 待审核记录列表
- 已审核记录列表
- 审核详情查看
- 审核通过操作
- 审核退回操作
- 重新提交操作
- 审核历史记录

### 1.3 与其他模块的关联

| 关联模块 | 关联关系 | 说明 |
|----------|----------|------|
| 成本分析 | 流程关联 | 成本分析提交后进入审核流程 |
| 标准成本 | 数据关联 | 审核通过可设为标准成本 |
| 通知中心 | 通知关联 | 审核结果产生通知 |

## 二、审核状态机

### 2.1 状态定义

| 状态 | 代码 | 说明 |
|------|------|------|
| 草稿 | draft | 创建后未提交的状态 |
| 已提交 | submitted | 已提交待审核的状态 |
| 审核通过 | approved | 审核员审核通过的状态 |
| 已退回 | rejected | 审核员退回的状态 |

### 2.2 状态流转图

```
┌─────────┐    提交      ┌───────────┐    通过     ┌──────────┐
│  draft  │ ───────────> │ submitted │ ──────────> │ approved │
│ (草稿)  │              │ (待审核)   │             │ (已通过)  │
└────┬────┘              └─────┬─────┘             └──────────┘
     │                         │
     │  重新提交                │ 退回
     └─────────────────────────┘
              rejected (已退回)
```

### 2.3 状态转换规则

| 当前状态 | 目标状态 | 触发操作 | 权限要求 |
|----------|----------|----------|----------|
| 草稿 | 已提交 | 提交审核 | 创建人 |
| 已退回 | 已提交 | 重新提交 | 创建人 |
| 已提交 | 审核通过 | 审核通过 | 审核员/管理员 |
| 已提交 | 已退回 | 审核退回 | 审核员/管理员 |

## 三、待审核记录

### 3.1 页面定位

待审核记录页面展示所有处于"已提交"状态的成本分析，审核员可以在此页面进行审核操作，业务员可以查看自己提交的成本分析状态。

1. 加载待审核列表数据（后端分页）
2. 加载用户权限信息
3. 根据权限显示不同的操作按钮

- **搜索范围**：成本分析编号、客户名称、型号

| 产品型号 | 具体型号名称 | - |

### 3.4 审核详情弹窗

#### 3.4.1 弹窗结构

- **左侧**：成本分析详情（基本信息、成本明细）
- **右侧**：审核操作面板

#### 3.4.2 详情展示

- 基本信息卡片（编号、客户、型号、数量等）
- 运费信息卡片（外销时显示）
- 成本明细标签页（原料、工序、包材）
- 成本汇总面板

#### 3.4.3 审核操作

**审核通过**：
- 按钮样式：主按钮（绿色）
- 操作：点击后弹出确认对话框
- 确认后：状态变为"审核通过"，记录审核历史

**审核退回**：
- 按钮样式：危险按钮（红色）
- 操作：点击后显示退回原因输入框
- 必填项：退回原因
- 确认后：状态变为"已退回"，记录审核历史和退回原因

**批注**：
- 输入框：多行文本
- 说明：可选，用于记录审核意见

#### 3.5.1 整体布局

#### 3.5.2 移动端适配

## 五、审核业务规则

### 5.1 权限规则

| 操作 | 权限要求 | 特殊限制 |
|------|----------|----------|
| 查看待审核列表 | review:view | 业务员只能看自己的 |
| 查看已审核列表 | review:view | 业务员只能看自己的 |
| 执行审核 | review:approve | 不能审核自己的 |
| 删除记录 | cost:delete:all | 仅管理员 |

### 5.2 自审限制

- **规则**：审核员不能审核自己创建的成本分析
- **实现**：审核时检查创建人是否为当前用户
- **提示**："不能审核自己创建的成本分析"

### 5.3 并发控制

#### 5.3.1 问题场景

**场景描述**：
两个审核员（审核员A和审核员B）同时打开同一个待审核的成本分析详情页面，几乎同时点击"审核通过"按钮。

**可能的问题**：
1. 两人都看到状态为"已提交"
2. 两人都执行了审核通过操作
3. 导致数据不一致或重复审核

#### 5.3.2 解决方案

**数据库行级锁（FOR UPDATE）**：

使用 PostgreSQL 的 `FOR UPDATE` 子句对记录加排他锁，确保同一时间只有一个事务可以修改该记录。

**实现流程**：

```
事务开始
    │
    ▼
获取数据库连接
    │
    ▼
BEGIN（开启事务）
    │
    ▼
SELECT * FROM quotations WHERE id = ? FOR UPDATE
    │           │
    │           └─► 对指定记录加排他锁
    │               其他事务阻塞等待
    ▼
检查当前状态
    │
    ├── 状态为"已提交" ──► 执行审核操作
    │                         │
    │                         ▼
    │                      UPDATE quotations
    │                      SET status = 'approved',
    │                          reviewer_id = ?,
    │                          reviewed_at = NOW()
    │                      WHERE id = ?
    │                         │
    │                         ▼
    │                      INSERT INTO review_history
    │                      (quotation_id, action, operator_id, comment)
    │                      VALUES (?, 'approved', ?, ?)
    │                         │
    │                         ▼
    │                      COMMIT（提交事务，释放锁）
    │                         │
    │                         ▼
    │                      返回成功
    │
    └── 状态不为"已提交" ──► ROLLBACK（回滚事务）
                              │
                              ▼
                           返回错误："该成本分析已被其他审核员处理"
```

#### 5.3.3 代码实现示例

**后端实现逻辑**：

1. 使用 `pool.connect()` 获取数据库客户端连接
2. 开启事务（BEGIN）
3. 使用 `FOR UPDATE` 锁定记录
4. 检查当前状态
5. 根据检查结果执行更新或回滚
6. 在 `finally` 块中释放连接

**关键要点**：
- 必须使用 `pool.connect()` 获取独立连接，不能使用连接池的自动管理
- 锁只在事务期间有效，事务提交或回滚后自动释放
- 其他事务等待锁释放时，PostgreSQL 默认会阻塞等待
- 可以设置锁等待超时，避免无限等待

#### 5.3.4 失败处理

**前端处理**：

当并发冲突发生时，后端返回特定错误码，前端根据错误码显示友好提示。

**错误提示**：
- 标题："审核失败"
- 内容："该成本分析已被其他审核员处理，请刷新列表查看最新状态"
- 按钮："确定"（点击后刷新列表）

**用户体验优化**：
1. 审核按钮点击后立即显示loading状态，防止重复点击
2. 审核成功后自动刷新待审核列表
3. 审核失败时提示用户刷新页面

#### 5.3.5 其他并发场景

**重新提交场景**：
- 业务员重新提交已退回的成本分析时
- 防止业务员A和业务员B（管理员）同时操作

**编辑场景**：
- 多人同时编辑草稿状态时
- 使用乐观锁（version字段）处理

**删除场景**：
- 管理员删除成本分析时
- 防止删除和审核同时进行

### 5.4 审核历史记录

- **记录内容**：审核动作（通过/退回）、审核人、审核时间、批注
- **存储方式**：独立的审核历史表
- **查询**：可在详情中查看完整审核历史

### 5.5 通知机制

- **审核通过通知**：通知创建人成本分析已通过
- **审核退回通知**：通知创建人成本分析被退回，包含退回原因
- **通知方式**：系统通知中心 + 仪表盘通知区域

## 六、重新提交流程

### 6.1 触发条件

- 成本分析状态为"已退回"
- 当前用户是创建人或管理员

### 6.2 操作流程

1. 在成本记录列表找到已退回的记录
2. 点击"编辑"按钮
3. 根据退回原因修改成本分析
4. 顶部显示退回原因警告提示
5. 修改完成后点击"提交审核"
6. 状态变为"已提交"，重新进入审核流程

### 6.3 退回原因展示

- **编辑页面**：顶部显示警告框，展示退回原因
- **详情页面**：在审核信息区域显示退回原因
